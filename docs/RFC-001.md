# RFC-001 — Keyboard Typing Trainer: Follow-ups

**Date:** 2026-06-07
**Status:** Draft — for discussion, not committed to implementation

---

## Context

The app is functionally complete per the PRD (steps 1–5). Step 6 (Polish Catppuccin Mocha) was deemed unnecessary — the theme is already applied consistently, transitions exist, and no visual issues were identified.

This RFC documents potential improvements for future consideration.

---

## 1. Majuscules / Shift

**Problem:** All drills and training text are lowercase. Users cannot practice capitalization (Shift + letter), which is essential for real typing (sentence starts, proper nouns, etc.).

**Approach:**
- Add Shift handling in `engine.js` — listen for Shift key state, produce uppercase when Shift is held
- Add Shift key to the visual keyboard (left/right Shift keys highlighted when a capital letter is expected)
- Generate drills with capital letters (e.g., `Ff Ff Jj Hh`)
- Training text: capitalize first word of each pseudo-sentence

**Complexity:** Medium — engine change + keyboard display change + drill generation change

---

## 2. Ponctuation dans les drills

**Problem:** Levels 6–8 include punctuation keys (`;`, `,`, `.`, `/`) in their key sets, but the drill generator never produces these characters. Training text also lacks punctuation (words are separated by spaces only).

**Approach:**
- Extend `generateDrill()` to include punctuation characters in the pool for levels that have them
- Training text: add commas and periods at sentence boundaries (e.g., `the cat sat, the dog ran.`)

**Complexity:** Low — small changes to drill generation logic

---

## 3. Rangée numérique

**Problem:** The number row is displayed on the visual keyboard but not included in any level's key set. Users cannot practice typing numbers.

**Approach:**
- Add levels 9+ for the number row, or extend levels 6–8 to include digits
- Alternatively: add a dedicated "Numbers" level after the full keyboard level

**Complexity:** Low — add level specs + digits in drill generation. Keyboard display already shows the row.

**Open question:** The PRD says "zone alphanumérique uniquement" — do numbers belong in the scope?

---

## 4. Responsive design

**Problem:** No media queries. Fixed `max-width: 800px`. On narrow screens (mobile, cmux sidebar), the keyboard visual and text display may overflow or be cramped.

**Approach:**
- Add breakpoints for widths < 600px: smaller font, compact keyboard keys, reduced padding
- Test on mobile viewport sizes

**Complexity:** Low — CSS only, no JS changes

---

## 5. Indicateur de doigt actif

**Problem:** The keyboard highlights the next key, but doesn't explicitly show which finger to use. Beginners may not intuitively map the color to a finger.

**Approach:**
- When a key is highlighted, show the finger name below it (e.g., "Left Index" or "L.Index")
- Or: show a hand diagram that highlights the active finger
- Or: add a tooltip on hover over keyboard keys showing the finger assignment

**Complexity:** Low–Medium depending on approach

---

## 6. Historique de progression

**Problem:** Scores are saved in localStorage but never displayed. Users have no way to review their progress over time (best CPM per level, improvement trend).

**Approach:**
- Add a "Stats" screen or panel showing: completed levels, best CPM/success per level, total practice time
- Could be a modal or a dedicated tab
- Data is already in the store — just needs a UI to display it

**Complexity:** Medium — new UI component + data aggregation

---

## 7. Replay à tout moment

**Problem:** Enter only relaunches a drill when the current drill is finished. There's no way to restart mid-drill (e.g., if you want a fresh attempt on the same level).

**Approach:**
- Add a keyboard shortcut (e.g., Ctrl+Enter or Esc) to restart the current drill at any time
- Or: add a visible "Replay" button in the text display area

**Complexity:** Low — add a key handler in the existing keyboard capture

---

## 8. Corpus plus riche

**Problem:** The French corpus has ~400 words, many filtered out by accent requirements. AZERTY levels 5–6 (home row + top row) have limited French word variety. English corpus is better (~800 words) but could still feel repetitive at higher levels.

**Approach:**
- Expand French corpus to ~1000+ words, including more accent-free words for QWERTY compatibility
- Expand English corpus to ~1500+ words with more 5–6 letter words for variety
- Consider adding common short phrases or collocations for more natural text

**Complexity:** Low — just more data in the word files, no code changes

---

## 9. Accents en QWERTY français

**Problem:** French words with accented characters (é, è, ç, à, ù) only appear when AZERTY layout is active. A French user on QWERTY will see fewer French words in training mode, and may be surprised that common words are missing.

**Approach:**
- Option A: Accept this as correct behavior — accents aren't on QWERTY, so accented words can't be practiced
- Option B: Add an accent-stripping variant of the French corpus (e.g., "etre" instead of "être") for QWERTY users
- Option C: Show a notice when French is selected with QWERTY: "Accented words unavailable on this layout"

**Complexity:** Very low for option A (no change) or C (UI message). Medium for B (alternate corpus).

---

## 10. Traduction complète de l'interface

**Problem:** The language toggle only affects the training text vocabulary (French vs English words). The UI itself is a mix of French and English with no i18n system:
- Level names: English ("Home — Index + Thumb")
- KPI labels: English ("Speed", "Success")
- Completion messages: French ("Terminé", "Appuie sur Entrée")
- Mode tabs: French ("Apprentissage", "Entraînement")
- Picker menus: English ("English", "Français", "QWERTY", "AZERTY")

When a user switches to French vocabulary, they expect the entire UI to switch to French, and vice versa.

**Approach:**
- Add a simple i18n system: a `strings` object keyed by language, with entries for every UI text
- Apply the current language to all static text (labels, messages, tooltips, tab names)
- Keep layout names (QWERTY/AZERTY) and level numbers as-is — they're universal
- Level names should be translated (e.g., "Home — Index + Thumb" → "Base — Index + Pouce")

**Complexity:** Medium — systematic but not architecturally complex. Every hardcoded string needs to be extracted and replaced with a lookup.

---

## 11. Menu de configuration

**Problem:** Some app parameters are hardcoded with no way for the user to change them. Currently, the KPI validation thresholds (50 chars, 90% success, 20 CPM) are constants in `levels.js`. A fast typist may find 20 CPM too easy; a beginner may find 90% success too strict. No UI exists to adjust these.

Future configurable settings could include: drill length, font size, etc.

**Approach:**
- Add a burger menu (hamburger icon) in the header, next to the mode tabs
- Opens a settings panel/dropdown with configurable options
- First setting: KPI thresholds (min chars, min success %, min CPM) with number inputs
- Settings persisted via `store.js` (already has the infrastructure)
- `THRESHOLDS` in `levels.js` reads from store instead of being hardcoded constants

**Complexity:** Low–Medium — UI component + store integration. The settings schema is small initially.

---

## 12. Pause d'un drill

**Problem:** Currently there is no way to pause a drill. If the user stops typing, the CPM keeps decaying (elapsed time grows while correctKeystrokes stay fixed). This was partially addressed by freezing the timer on drill completion (`finishTime`), but during the drill itself, any pause (distraction, break) penalizes the CPM.

**Approach:**
- Add a pause/resume toggle (e.g., Esc key, or a pause button)
- When paused: `engine.getStats()` uses a saved `pausedElapsed` instead of `Date.now() - startTime`, so CPM doesn't decay
- When paused: keyboard input is ignored, visual indicator shows "Paused"
- On resume: adjust `startTime` by the pause duration so the timer continues from where it left off

**Complexity:** Low — engine change (pause state + elapsed adjustment) + UI indicator

---

## 13. Progression dans le drill

**Problem:** The user has no visual indication of how far they are in the current drill (150 characters). They only see the text and the cursor, but no sense of overall progress. This makes long drills feel like they go on forever.

**Approach:**
- Add a thin progress bar at the top of the `#text-display` container
- Width = `(position / text.length) * 100%`
- Color: Catppuccin Mocha mauve (consistent with the cursor)
- Or: show a fraction counter like `42/150` near the stats

**Complexity:** Very low — one CSS element + one line in `ui.js` to update its width on each render

---

## 14. Mode libre — texte personnalisé

**Problem:** The app only generates text from drills (random character patterns) or from the word corpus (filtered by level keys). There is no way for the user to type arbitrary text of their choice — a favorite quote, a specific sentence they want to practice, or code snippets.

**Approach:**
- Add a third mode tab: "Libre" (Free) alongside Apprentissage and Entraînement
- In free mode: show a text input where the user types or pastes the text they want to practice
- On confirm (Enter or a "Start" button), the engine loads that text and typing begins
- No level selection, no KPI thresholds — pure practice
- Keyboard visual still highlights the next key
- Stats still run (CPM, success rate) but no completion validation

**Complexity:** Low — new mode tab + a simple text input UI + wire into existing engine

---

## 15. Niveaux configurables

**Problem:** Levels are hardcoded in `levels.js` as a fixed array of 8 specs. Users cannot create custom levels, reorder them, or adjust their progression. A typing teacher might want to define specific drills for their students; a user might want to focus on specific weak keys.

**Approach:**
- Define levels as an editable JSON structure (stored in `store.js`), with fields:
  - `title` — short name (e.g., "Home Row")
  - `description` — longer explanation (e.g., "Practice all keys on the home row")
  - `prompt` — instruction for text generation (e.g., "Generate a drill using keys: a, s, d, f, g, h, j, k, l")
  - `keys` — explicit key set (optional, can be derived from prompt)
- Ship a default config matching the current 8 levels
- Add a level editor accessible from the settings menu (#11):
  - Add/edit/delete/reorder levels
  - Each level editable as a simple form (title, description, prompt, keys)
- Import/export config as JSON file (download/upload) for sharing between devices or users
- The app falls back to default levels if no custom config exists

**Complexity:** Medium — level editor UI + JSON schema + import/export + refactor `levels.js` to read from config instead of hardcoded specs. The text generation logic would need to accept a prompt/key-set instead of a level ID.

---

## 16. Suivi des scores dans le temps

**Problem:** Currently the app only stores the best score per level (single CPM + success rate entry in `scores`). There is no history, no trend tracking, no way to see if you're improving over time. The user gets no sense of long-term progress.

**Approach:**
- Record a score entry after each completed drill: `{ date, level, mode, cpm, successRate, layout }`
- Store as an array in `store.js` (append-only log)
- Add a stats/history screen (accessible from settings menu #11 or a dedicated tab):
  - **Per-level view**: chart of CPM and success rate over time for a selected level
  - **Global view**: overall CPM trend across all levels, total practice time, total drills completed
  - **Summary cards**: best CPM ever, current streak, levels completed
- Visualization: simple line charts (could use a lightweight chart library, or render with canvas/SVG — no external dependencies per project constraints)
- Pruning: keep last 100 entries per level, or last 30 days, to prevent localStorage bloat

**Complexity:** Medium — score logging is trivial (store change), but the visualization UI is the bulk of the work. Without external chart libraries, SVG/canvas rendering is needed. This subsumes and extends follow-up #6 (Historique de progression).
