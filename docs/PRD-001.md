# PRD-001 вЂ” Mode Libre

**Feature:** Free typing mode (RFC-001 В§14)
**Source:** [RFC-001](RFC-001.md) вЂ” Follow-up #14: Mode libre вЂ” texte personnalisГ©
**Branch:** `feat/free-mode`
**Date:** 2026-06-07

---

## Context

The app offers two modes today: **Apprentissage** (drills with level progression + validation thresholds) and **EntraГ®nement** (real words filtered by level keys, no validation). Both generate text automatically вЂ” the user never chooses what they type.

The RFC identifies a gap: there is no way to type **arbitrary text** вЂ” a favorite quote, a specific sentence, code snippets, or any text the user wants to practice. Follow-up #14 ([`RFC-001.md`](RFC-001.md)) proposes a third mode, "Libre" (Free), to fill this gap.

---

## Problem

- Users cannot practice on text they chose themselves (quotes, code, tricky sentences).
- The two existing modes limit variety to generated drills and filtered word corpora.
- No way to rehearse a specific passage before typing it "for real" (e.g. a password, a command, an email draft).

---

## Solution

Add a third mode tab **"Libre"** alongside Apprentissage and EntraГ®nement.

### Core flow

1. User clicks "Libre" в†’ the text-display area becomes an editable textarea where they type or paste their custom text.
2. User clicks **"Commencer В· Ctrl+Enter"** (or presses **Ctrl+Enter**) в†’ the engine loads that text, the textarea reverts to the standard drill view, and typing begins.
3. While typing: keyboard visual highlights the next key (when it exists on the layout), stats run normally (CPM + success rate), no KPI thresholds.
4. On completion: "TerminГ©" screen with hint "EntrГ©e = recommencer В· Esc = modifier le texte". Enter replays the same text. Esc or "Modifier le texte" button returns to the input phase.

### Design decisions

| # | Decision | Rationale |
|---|---|---|
| Q1 | Disable global `#input-capture` re-focus during input phase; visible textarea takes focus | Both textareas can't share focus. Switch back to `#input-capture` on drill start. |
| Q2 | Action buttons ("Commencer" / "Modifier le texte") live in `#level-bar` area | Level bar is hidden in Libre mode вЂ” reuses freed space. Keeps `#text-display` pure. |
| Q3 | Esc during drill = return to input phase, restart on "Commencer" | Esc is for editing text, not for pause/resume. Restart is clean вЂ” no cursor misalignment. Pause is a separate concern (RFC В§12). |
| Q4 | After Esc в†’ edit в†’ "Commencer": always restart from scratch | Resume after edit is fragile (cursor misalignment). User chose to edit = chose to start over. |
| Q5 | "TerminГ©" hint: "EntrГ©e = recommencer В· Esc = modifier le texte" | Accurate wording for Libre mode. Teaches the Esc affordance. |
| Q6 | Untypeable characters: accept вЂ” user types any key, it's an error, cursor advances | Engine already advances on errors. User is never stuck. No skip mechanism needed. |
| Q7 | Textarea styled to match drill view exactly | Visual continuity вЂ” same font, size, padding, height, background. No resize handle. Height auto-grows within container. |
| Q8 | Last custom text persisted in `store.js` as `lastFreeText` | Survives page refresh. Trivial cost (one key). No history вЂ” just the last text. |
| Q9 | Switching away from Libre mid-drill: immediate, no confirmation | Stakes are zero. Text is saved in store. Consistent with existing mode-switch behavior. |
| Q10 | Returning to Libre: always input phase | Consistent with Q9 (drill is abandoned). User must consciously start. |
| Q11 | Drill-phase button label: "Modifier le texte" | Accurate вЂ” textarea is pre-filled. If user wants fresh text, they select-all + delete. |
| Q12 | Stats section hidden during input phase | Showing 0 CPM / 100% is misleading. Stats appear on drill start. |
| Q13 | Keyboard display stays visible but idle during input phase | No highlight = natural "no drill" signal. Avoids layout shift. Gives user context on which keys they'll need. |
| Q14 | Textarea: fixed 7.5rem height with internal scroll | Same size as drill view. Zero layout shift on transition. User types line by line anyway. |
| Q15 | Ctrl+Enter surfaced as secondary label on "Commencer" button | Typing app = keyboard-oriented users. Pattern: "Commencer В· Ctrl+Enter". |
| Q16 | One action button element in `#level-bar`, label changes by phase | Single element = single responsibility slot. Branch on state in click handler. |
| Q17 | Text preprocessed with `.trim()` before engine | Prevents typing invisible leading/trailing whitespace. Internal whitespace preserved. |
| Q18 | Tab label: "Libre" | Consistent with French single-word tabs. Short, maps cleanly to "Free" for i18n. |

### Detailed requirements

#### Text input phase

| Requirement | Detail |
|---|---|
| Input area | The `#text-display` container transforms into an editable `<textarea>` вЂ” same visual container, same size, same styling (font, padding, background, height 7.5rem, scrollbar hidden). |
| Focus | `#input-capture` global re-focus is **disabled**. The visible textarea receives focus naturally. |
| Placeholder | Grey placeholder text: "Tape ou colle ton texte iciвЂ¦" |
| Multi-line | Enter inserts a real newline. Fixed height with internal scroll вЂ” same as drill view. |
| Validation | Minimum **5 characters** (after `.trim()`) to start. "Commencer" button is disabled below this threshold. |
| Confirm | "Commencer В· Ctrl+Enter" button OR **Ctrl+Enter** keyboard shortcut. |
| Cancel | Switching to another mode tab discards the drill state (text preserved in store). |
| Wrapping | No automatic `\n` insertion. Textarea soft-wraps naturally. Explicit `\n` only from Enter key. Wrapping mismatch with drill view is acceptable вЂ” same container, font, padding ensures close alignment. |

#### Drill phase

| Requirement | Detail |
|---|---|
| Focus | `#input-capture` re-focus is re-enabled. Visible textarea is hidden. |
| Engine | Standard engine behavior. Newlines typed as Enter в†’ `\n`, displayed as `в†µ` / `<br>` per existing logic. Text is `.trim()`-ed. |
| Keyboard visual | Highlights the next key **if it exists** on the current layout. Characters not on the visual keyboard are typed without highlight вЂ” cursor simply advances. Keyboard stays visible and idle during input phase, active during drill phase. |
| Stats | CPM + success rate displayed as usual. Hidden during input phase, visible during drill phase. No thresholds, no level completion logic. |
| Finish | "TerminГ©" screen with frozen stats. Hint: "EntrГ©e = recommencer В· Esc = modifier le texte". |
| Replay | Enter replays the **same** text (same as current behavior). |
| Edit text | "Modifier le texte" button (in `#level-bar` area) or **Esc** key в†’ returns to input phase with textarea pre-filled. Restart on "Commencer". |

#### Mode-specific UI

| Element | Behavior in Libre mode |
|---|---|
| Level bar (`#level-bar`) | **Hidden** in both phases. Replaced by single action button ("Commencer В· Ctrl+Enter" or "Modifier le texte"). |
| Language picker | **Hidden** вЂ” not relevant (user provides their own text). |
| Layout picker | **Visible** вЂ” still affects the visual keyboard. |
| Mode tabs | Three tabs: Apprentissage / EntraГ®nement / Libre. Libre has the same visual treatment as the others (mauve when active). |
| Stats (`#stats`) | **Hidden** during input phase. **Visible** during drill phase. |
| Keyboard display | **Visible** in both phases. Idle (no highlight) during input phase. Active (highlight on next key) during drill phase. |

#### Mode transitions

| From в†’ To | Behavior |
|---|---|
| Any mode в†’ Libre | Level bar hides. Stats hide. Text-display shows input textarea (empty on first visit, pre-filled with `lastFreeText` from store on return). Keyboard display goes idle. Action button: "Commencer В· Ctrl+Enter". |
| Libre в†’ Any mode | Drill abandoned immediately. Text saved to `lastFreeText` in store. Level bar reappears. Stats reappear. Resumes previous level in the target mode. |
| Libre input в†’ Libre drill | Text `.trim()`-ed в†’ engine. `#input-capture` re-focus enabled. Textarea hidden. Stats shown. Keyboard display highlights. Action button: "Modifier le texte". |
| Libre drill в†’ Libre input | Engine resets. `#input-capture` re-focus disabled. Textarea shown pre-filled with `lastFreeText`. Stats hidden. Keyboard display goes idle. Action button: "Commencer В· Ctrl+Enter". |

## Implementation steps

Four vertical slices, strictly sequential. Each step produces a testable app.

### Step 1 вЂ” Libre mode shell + visibility toggling

The skeleton: the "Libre" tab exists, UI elements hide/show correctly, but no text input yet.

| What | Detail |
|---|---|
| Tab | Add "Libre" button in `#mode-tabs` (third tab, same styling as the others) |
| State | Add `mode === 'free'` + `freePhase: 'input' | 'drill'` on the App instance |
| Level bar | **Hidden** when `mode === 'free'`. Replaced by action button container in the same `#level-bar` area. |
| Language picker | **Hidden** when `mode === 'free'`. |
| Stats (`#stats`) | **Hidden** when `mode === 'free'` and `freePhase === 'input'`. |
| Keyboard display | **Idle** (no highlight) when `freePhase === 'input'`. |
| Action button | "Commencer В· Ctrl+Enter" in `#level-bar` area вЂ” **disabled** (no textarea yet to validate against). |
| Text-display | Placeholder text "Tape ou colle ton texte iciвЂ¦" (greyed, not yet a textarea). |
| Mode switch | Clicking Apprentissage/EntraГ®nement restores level bar, language picker, stats, keyboard highlights. |

рџ‘‰ **Testable:** click Libre в†’ level bar gone, language picker gone, stats gone, keyboard idle, placeholder visible, disabled button. Click Apprentissage в†’ everything restores.

### Step 2 вЂ” Text input phase

The textarea works: type/paste, validation, shortcut.

| What | Detail |
|---|---|
| Textarea | Replace `#text-display` content with an editable `<textarea>` when `mode === 'free'` and `freePhase === 'input'`. Same CSS as drill view (font, padding, background, height 7.5rem, scrollbar hidden, no resize handle). |
| Focus | Disable global `#input-capture` re-focus (`document.addEventListener('click')`). Visible textarea receives focus on mount and on click. |
| Placeholder | `placeholder="Tape ou colle ton texte iciвЂ¦"` on the textarea. |
| Validation | Enable/disable "Commencer" button based on `textarea.value.trim().length >= 5`. |
| Ctrl+Enter | Listen for `keydown` on the textarea. If `e.ctrlKey && e.key === 'Enter'`, trigger the same action as the "Commencer" button. |
| Phase transition guard | Nothing happens on "Commencer" yet (Step 3 wires the engine). Button enables/disables but click/Ctrl+Enter is a no-op for now, or shows a console log. |

рџ‘‰ **Testable:** type/paste text in textarea, see button activate at 5 chars, press Ctrl+Enter. Focus stays on textarea (not stolen by `#input-capture`).

### Step 3 вЂ” Drill phase: the complete loop

The core вЂ” input в†’ drill в†’ TerminГ© в†’ replay.

| What | Detail |
|---|---|
| Start drill | On "Commencer" click or Ctrl+Enter: `.trim()` the textarea value в†’ `engine.reset(trimmedText)`. Set `freePhase = 'drill'`. |
| Phase switch в†’ drill | Hide textarea. Show `#text-display` as standard drill view (TextDisplay.render). Re-enable `#input-capture` re-focus. Show stats. Keyboard display becomes active (highlight on next key). Action button в†’ "Modifier le texte". |
| TerminГ© screen | Custom hint: "EntrГ©e = recommencer В· Esc = modifier le texte". (Modify `TextDisplay._renderFinished()` to accept a custom hint based on mode.) |
| Replay | Enter at end в†’ `app.replayDrill()` replays the **same** text (engine.reset with same text). |
| Untypeable chars | No special handling вЂ” engine already advances on error. Keyboard highlight: `KeyboardDisplay` only highlights keys that exist on the layout. |

рџ‘‰ **Testable:** paste a quote, click Commencer, type the drill, see TerminГ© with custom hint, Enter to replay. Full cycle.

### Step 4 вЂ” Esc, "Modifier le texte", and persistence

Return paths and persistence вЂ” closes the loop.

| What | Detail |
|---|---|
| Esc during drill | On `keydown` Esc when `freePhase === 'drill'`: set `freePhase = 'input'`, show textarea pre-filled with the drill text. |
| "Modifier le texte" button | Click handler: same as Esc вЂ” switch to input phase, pre-fill textarea. |
| Store: `lastFreeText` | On "Commencer" (Step 3): `store.set('lastFreeText', trimmedText)`. |
| Return to Libre | When switching to Libre mode: load `lastFreeText` from store. If exists в†’ pre-fill textarea. If not в†’ empty textarea with placeholder. Always `freePhase = 'input'`. |
| Page refresh | `lastFreeText` in localStorage survives refresh. On app boot, if `mode === 'free'` в†’ restore from store. |
| Mode switch guard | When leaving Libre (any mode switch): current text saved to `lastFreeText`. On return в†’ pre-filled. |

рџ‘‰ **Testable:** drill mid-way в†’ Esc в†’ textarea pre-filled в†’ edit в†’ restart. Switch to Apprentissage в†’ back to Libre в†’ text there. Refresh page в†’ text there.

## Out of scope

- **Text history / multiple saved texts** вЂ” only the last text is persisted. No history list or save slots.
- **Import from file** вЂ” paste is sufficient for v1.
- **Text length limit** вЂ” no hard cap in v1. The engine handles any length.
- **KPI thresholds** вЂ” explicitly none. Libre mode is pure practice, no validation.
- **Pause/resume** вЂ” belongs to RFC В§12 as a separate feature. Esc in Libre mode is for editing text, not for pause.
- **Auto `\n` insertion at wrap points** вЂ” wrapping is left to the browser. Soft-wrap mismatch between textarea and drill view is accepted.

---

## Success criteria

1. User can switch to Libre mode, type/paste text, start a drill, and type it with full engine + visual keyboard + stats support.
2. Newlines in custom text are preserved and typeable (Enter = `\n` during drill).
3. Characters not on the visual keyboard don't break anything вЂ” cursor advances, no highlight, counts as error.
4. Replay works (Enter at end). Return-to-edit works (Esc or "Modifier le texte" button).
5. Switching away from Libre mode and back shows the input phase with the last entered text pre-filled (persisted in store).
6. Level bar and language picker are hidden in Libre mode; layout picker stays visible.
7. Stats are hidden during input phase, visible during drill phase.
8. Keyboard display stays visible in both phases (idle during input, active during drill).
9. Ctrl+Enter starts the drill from the input phase. The shortcut is surfaced on the "Commencer" button.
10. Text is `.trim()`-ed before being sent to the engine.
